# 매칭 요청 제한 정책 변경 — 프론트 대응 가이드

## 정책 변경 요약

기존: 프로필에 엮인 매칭이 **하나라도** 있으면(받거나 걸거나 무관) 다른 매칭이 불가능.

변경 후:

- 매칭 요청 **걸기는 한 사람한테만** (보낸 요청 1건 제한)
- 매칭 요청 **받기는 다수 가능**
- **받은 요청 여부와 무관하게** 요청 걸기 가능
- **성사되면** 해당 두 프로필에 엮인 다른 모든 활성 요청이 자동 취소(`CANCELLED`)

### 동작 예시

| 시나리오 | 결과 |
|---|---|
| `A→B` 있는데 `A→C` 시도 | ❌ 차단 (걸기 1건 제한) |
| `A→B` 있는데 `B→C` 시도 | ✅ 허용 (B는 받기만 했으므로 걸기 가능) |
| `A→C` 있는데 `B→C` 시도 | ✅ 허용 (C는 여러 요청 받기 가능) |
| `A→C` 성사 | A·C에 엮인 다른 모든 PENDING 요청 자동 취소 |

> `A→B` = A의 친구 프로필이 B의 친구 프로필에게 요청. 노드는 모두 **친구 프로필** 단위.

---

## 프론트 필수 작업

### 1. 새 알림 타입 `match_cancelled` 처리

- 알림 목록(`GET /notifications`)의 `type` 필드에 `"match_cancelled"`가 새로 내려갑니다.
- 분기 처리가 없으면 알림이 빈칸/에러로 표시될 수 있으니 **문구 매핑 추가** 필요.
  - 예: "상대가 다른 분과 매칭되어 요청이 취소되었습니다."
- 이 알림의 `matchingId`로 상세를 열면 해당 매칭은 `CANCELLED` 상태.

### 2. 걸기 1건 제한 — 새 409 에러 처리

- `POST /matchings` 요청 시, 내 친구 프로필이 이미 다른 곳에 요청을 보낸(PENDING) 상태면:
  - HTTP `409 CONFLICT`
  - 메시지: `"내 친구 프로필이 이미 다른 매칭 요청을 보낸 상태입니다. 한 번에 한 명에게만 요청할 수 있습니다."`
- 가능하면 **버튼 단계에서 사전 차단** 권장:
  - 이미 보낸 PENDING 요청이 있는 친구 프로필은 "요청 보내기" 비활성화.
  - 판단 근거: 보낸 요청 목록 `GET /matchings/sent`에서 `status == PENDING` 존재 여부.

---

## 동작 변경 — 확인 필요

### 3. 카드/공개목록에 "요청받은 프로필"이 계속 노출됨

- 기존엔 누군가 요청만 걸어도 그 프로필이 카드/공개목록에서 사라졌으나, 이제는 **실제 성사 전까지 계속 노출**됩니다. (여러 명이 요청 가능해야 하므로 의도된 변경)
- 프론트가 "이미 내가 요청 보낸 프로필"을 카드에서 구분하던 로직이 있다면, 이제 그 프로필도 목록에 남으므로 **"요청 보냄" 뱃지 또는 버튼 비활성화** 처리 권장.

### 4. 성사 직후 다른 요청들이 `CANCELLED`로 전이됨

- 보낸/받은 요청 목록(`GET /matchings/sent`, `GET /matchings/received`)에서 화면에 떠 있던 PENDING 요청이 서버에서 `CANCELLED`로 바뀔 수 있습니다.
- 매칭 성사/알림 수신 시 목록을 **refetch**하거나, `CANCELLED` 상태를 "취소됨(상대 매칭 성사)"으로 표시 권장.

---

## 안 바꿔도 되는 것

- API 엔드포인트, 요청/응답 스키마(필드 구조) 그대로.
- `status` 값 종류(`PENDING` / `MATCHED` / `CANCELLED` 등) 기존 enum 그대로 — 새 상태값 없음.
