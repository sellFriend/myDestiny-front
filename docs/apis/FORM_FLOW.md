# 지인 등록 폼 접근 플로우 (FORM_FLOW)

내 친구(주선자)가 지인에게 **친구 등록 폼**을 보내고, 지인이 카카오 인증을 거쳐 폼을
작성·제출하기까지의 전체 흐름을 정리한 문서입니다.

> ⚠️ 이 문서는 **기획상 의도한 플로우**를 기준으로 작성되었습니다.
> 현재 백엔드 API 명세(`mydestiny-api-spec.pdf`)와 일치하지 않는 부분이 있으며,
> 맨 아래 [현재 API 명세와의 갭](#5-현재-api-명세와의-갭) 절에 차이를 정리해 두었습니다.

---

## 1. 핵심 개념

- **주선자**: 서비스에 가입한 사용자. 자신의 지인을 소개팅 후보로 등록한다.
- **지인**: 별도 계정이 없는 등록 대상. 폼 링크를 통해서만 정보를 입력한다.
- **숏링크 토큰(이하 `token`)**: 친구 등록 버튼을 누를 때 백엔드가 발급하는, 폼 링크에
  포함된 식별값. 이 링크에는 백엔드 응답에 기반한 특정 값이 담겨 있다.
- **인가코드(이하 `authCode`)**: 지인이 폼 접근을 위해 카카오 로그인을 했을 때 받는
  카카오 OAuth authorization code. **주선자로 이미 가입했는지 검증**하는 용도다.

폼에 접근하려면 **`token`과 `authCode` 두 값이 모두** 있어야 한다.

---

## 2. 전체 흐름

```
[주선자]
  내 친구 페이지 → "친구 등록" 버튼 클릭
        │
        ▼
  백엔드에 숏링크 생성 요청 → token이 포함된 폼 링크(shortlink) 발급
        │
        ▼
  링크를 지인에게 공유 (카카오톡 / 문자 / 인스타 DM / 복사)

[지인]
  공유받은 링크 접근
        │
        ▼
  카카오 로그인 (★ 인증 목적 전용 — 웹 세션 로그인 아님)
        │
        ▼
  카카오 인가코드(authCode) 수령
        │
        ▼
  ┌─────────────────────────────────────────────┐
  │ 백엔드가 authCode로 "이미 주선자로 가입된     │
  │ 유저인지" 검증                                │
  └─────────────────────────────────────────────┘
        │
   ┌────┴─────────────────────────┐
   ▼                              ▼
 [이미 주선자]                   [주선자 아님]
 "주선자는 등록할 수 없습니다"     token + authCode 보유 →
 폼 접근 차단                    폼 접근 허용
                                      │
                                      ▼
                                폼 전체 입력 후 제출
                                (제출 시 token + authCode 동봉)
```

---

## 3. 단계별 상세

### 3-1. 숏링크 생성 (주선자)

- **위치**: 내 친구 페이지 → "친구 등록" 버튼
- **동작**: 백엔드에 숏링크 생성을 요청하고, 응답으로 받은 `token`이 포함된 폼 링크를
  생성한다. 이 링크는 단순 프론트 랜덤 토큰이 아니라, **백엔드가 발급한 값**을 담는다.
- **공유**: 생성된 링크를 카카오톡 / 문자 / 인스타 DM / 클립보드 복사로 공유한다.
  (웹은 토스트로 복사 안내, 앱은 바텀시트로 공유 채널 노출 — 이미 구현됨)

### 3-2. 링크 접근 + 카카오 로그인 (지인)

- 지인이 공유받은 링크에 접근하면 **카카오 로그인**을 먼저 거친다.
- 이 로그인의 목적은 **본인 인증 + 주선자 중복가입 여부 확인**이다.
- **중요 — 인증 전용 로그인**:
  - 여기서의 카카오 로그인은 **웹 서비스 로그인 처리로 이어지면 안 된다.**
  - 즉, 이 로그인 직후 홈으로 이동했을 때 **로그인된 상태로 보이면 안 된다.**
  - 일반 로그인과 토큰 저장소/세션을 **분리**하거나, 인증 코드만 일회성으로 사용하고
    세션을 만들지 않아야 한다.

### 3-3. 인가코드 검증 (백엔드)

- 받은 `authCode`로 백엔드가 **DB에 주선자로 가입된 유저인지** 확인한다.
- **이미 주선자인 경우**: `"주선자는 등록할 수 없습니다"` 문구를 노출하고 **폼 접근을 차단**한다.
- **주선자가 아닌 경우**: `token` + `authCode`를 보유한 채 폼 접근을 허용한다.

### 3-4. 폼 작성 및 제출 (지인)

- 폼 전체를 입력한 뒤 제출한다.
- **제출 시 `token`과 `authCode`를 함께 전송**한다. (둘 다 있어야 서버가 제출을 수락)

---

## 4. 프론트엔드 상태/라우팅 요구사항

| 요구사항 | 설명 |
| --- | --- |
| 폼 접근 가드 | `token`과 `authCode`가 모두 있어야 폼 렌더링. 하나라도 없으면 카카오 로그인 유도 또는 에러 |
| 인증 전용 로그인 | 폼 진입용 카카오 로그인은 전역 `AuthContext`/토큰 저장소에 **세션을 남기지 않는다** |
| 주선자 차단 화면 | 검증 결과 "이미 주선자"이면 `"주선자는 등록할 수 없습니다"` 안내 + 폼 차단 |
| 제출 payload | 폼 데이터 + `token` + `authCode`를 함께 서버로 전송 |

## 5. 현재 API 명세와의 갭

`mydestiny-api-spec.pdf` 기준으로, **숏링크 생성만 존재**하고 나머지 인가코드 기반
게이팅 흐름은 아직 정의되어 있지 않습니다.

### ✅ 명세에 있는 것

| 기능 | 엔드포인트 | 비고 |
| --- | --- | --- |
| 숏링크(폼 링크) 생성 | `POST /api/acquaintances/invite` | 응답 `{ formUrl, expiresAt }`, Auth 필요 |
| 폼 링크 유효성 확인 | `GET /form/{token}` | **인증 불필요(public)**, 토큰 유효성만 확인 |
| 폼 제출 | `POST /form/{token}` | body: name/age/phoneNumber/email/kakaoId/instagramId 등 |
| 폼 사진 업로드 | `POST /form/{token}/photos` | multipart |

### ❌ 명세에 없는 것 (기획 플로우에는 필요)

1. **인가코드 수령 흐름**
   - 명세의 카카오 OAuth는 raw 인가코드를 프론트가 받지 않는다.
   - 서버가 코드를 토큰으로 교환해 `{REDIRECT_URI}?accessToken=...&refreshToken=...`로
     리다이렉트한다. (프론트는 토큰만 본다)
2. **`token` + `authCode` 2개로 폼 접근** — `GET /form/{token}`은 토큰만으로 접근하며
   인가코드 파라미터가 없다.
3. **주선자 중복가입 검증 / "주선자는 등록할 수 없습니다" 차단** — 해당 검증 API/응답 필드 없음.
4. **인증 전용(세션 비로그인) 카카오 로그인** — OAuth 콜백은 항상 토큰을 발급하므로
   현재 구조상 곧 로그인 처리가 된다. 분리된 "인증 전용 모드"가 명세에 없다.
5. **폼 제출 시 인가코드 동봉** — `POST /form/{token}` body에 `authCode`/`token` 필드가 없다.

### 백엔드에 필요한 추가 요청(제안)

- 폼 접근 검증용 엔드포인트: `token` + 카카오 `authCode`(또는 인증 결과)를 받아
  `{ accessible: boolean, reason: "ALREADY_MADAM" | null }` 형태로 응답.
- 인가코드를 **세션 생성 없이** 검증만 하는 OAuth 모드(또는 `state`/`purpose=form_verify`
  파라미터로 콜백 분기).
- `POST /form/{token}` 요청 body 또는 헤더에 `authCode` 추가.
