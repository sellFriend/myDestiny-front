## 0. 한눈에 보기

**My Destiny**는 **지인 소개 기반(Acquaintance-based) 소개팅 매칭 플랫폼**이다.
당사자가 직접 자신을 어필하는 기존 데이팅 앱과 달리, **"나를 잘 아는 가까운 사람(주선자)이 대신 소개하고 연결해주는"** 구조를 택했다.

- **랜딩 카피**: "좋은 사람은, 가까운 사람이 더 잘 아니까."
- **핵심 역할**: **주선자** 가 **지인(소개 대상)** 을 등록하고, 다른 주선자의 지인 카드를 둘러보다 마음에 들면 **연결을 요청**한다. 양쪽 주선자가 동의하면 매칭이 성사되고 연락처가 공유된다.
- **기술 스택**: React 19 + TypeScript + Vite 7 + React Router 7 + TanStack Query 5 + Tailwind 4 + framer-motion + zustand. 카카오 OAuth2 + JWT 인증.

## 1. 배경 — 어떤 UX 고찰에서 출발했는가

### 1-1. 타겟과 Pain Point

| 대상                           | 기존 데이팅 서비스에서의 Pain Point                                                                                                    | My Destiny의 접근                                                                                        |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **소개받고 싶은 당사자**       | "내가 직접 나서서 불특정 다수에게 나를 어필·홍보하는 게 부담스럽다", "프로필을 매력적으로 꾸미는 것 자체가 일", "상대가 진짜인지 믿기 어렵다" | 당사자는 **앱에 가입할 필요가 없다.** 주선자가 보낸 **폼 숏링크로 본인이 직접 정보를 입력**하되, 그 뒤의 **소개·보증·카드 노출 관리는 주선자가 대신**한다(공개/비공개도 주선자가 조절). 연락처는 매칭 성사 전까지 공개되지 않는다. |
| **주선을 즐기는 사람(주선자)** | "주변에 좋은 친구 둘을 엮어주고 싶은데 마땅한 도구가 없다", "단톡방·수기 소개는 번거롭고 민망하다"                                     | 친구를 카드로 등록 → 다른 주선자의 카드를 탐색 → 연결 요청. **주선 행위 자체를 하나의 경험으로 제품화.** |
| **신뢰가 중요한 사용자**       | "랜덤 매칭은 검증되지 않은 사람이 많다"                                                                                                | **양쪽에 모두 사람(주선자)이 끼는 이중 검증 구조.** 아는 사람이 보증한 사람만 카드가 된다.               |

> 한 줄 요약 — **"불특정 다수에게 나를 직접 파는 부담"** 과 **"검증되지 않은 상대에 대한 불신"** 을, **지인이 대신 소개·보증하는 구조**로 동시에 해소한다.

### 1-2. 일관되게 적용한 UX 원칙

코드 곳곳과 `docs/UX`, `README`에 드러난 설계 철학:

- **점진적 공개(Progressive Disclosure)** — 폼 입력과 매칭 상세 모두 한 번에 다 보여주지 않고 단계적으로 드러낸다.
- **Hick's Law(선택지 축소)** — MBTI를 16지선다 대신 4개의 2지선다 축으로 분해하는 등 결정 부담을 낮춘다.
- **관계 중심 · 2인칭 · 안심 카피(토스 톤)** — 내부 용어(requester/target/주선자)를 화면에서 지우고 "OO님이 제안했어요" 같은 관계 문장으로 쓴다.
- **모노크롬 + 파스텔 포인트** — 흑백 기반 UI에 파스텔(라임/라일락/민트/코랄/크림/핑크) 카드 색으로 정체성을 준다.
- **모바일 우선** — 하단 탭 네비게이션, 드래그로 닫는 바텀시트 등 앱 같은 인터랙션.

## 2. 핵심 개념과 등장인물

| 용어                             | 정의                                                                                     |
| -------------------------------- | ---------------------------------------------------------------------------------------- |
| **주선자**                       | 카카오로 가입한 서비스 사용자. 지인을 등록하고 카드를 탐색하며 연결을 요청한다.          |
| **지인(소개 대상)**              | 별도 계정이 없는 등록 대상. 주선자가 보낸 **폼 숏링크**를 통해서만 자기 정보를 입력한다. |
| **폼 숏링크 (`/form/:madamId`)** | 주선자가 지인에게 보내는 등록 폼 링크. 끝의 UUID가 주선자 식별 코드.                     |
| **데이팅 프로필 / 카드**         | 지인의 정보가 주선자 승인을 거쳐 공개되면 탐색 화면에 노출되는 카드.                     |
| **매칭(요청)**                   | 한 주선자가 자기 지인을 상대 지인에게 소개하려고 보내는 연결 요청.                       |
| **당사자 동의(Consent)**         | 매칭 진행 중 양쪽 당사자(지인)의 동의 단계. (현재 화면 미구현, 백엔드 내부 단계)         |

### 전체 흐름 한 장 요약

```
[주선자 A]                                    [주선자 B]
  카카오 로그인                                카카오 로그인
     │                                          │
  내 친구 → "친구 추가" → 폼 숏링크 생성·공유        (동일)
     │                                          │
  [지인 a] 링크 접근 → 카카오 본인인증 → 폼 작성·제출    [지인 b] (동일)
     │                                          │
  주선자 A가 알림 받고 승인 → 지인 a 카드 공개           (동일)
     │                                          │
     └──────────► 탐색에서 서로의 카드를 둘러봄 ◄──────────┘
                          │
        주선자 A가 "지인 a를 지인 b에게 소개" 연결 요청 전송
                          │
              주선자 B가 받은 요청에서 수락
                          │
              (당사자 동의 단계 — 백엔드)
                          │
                   매칭 성사 → 양쪽 연락처 공유
```

## 3. 인증 구조 (모든 페이지의 전제)

### 3-1. 카카오 OAuth2 + JWT

- 로그인 시작: 프론트가 `GET {BASE_URL}/oauth2/authorization/kakao?redirect_uri={origin}/oauth2/callback` 로 이동.
- 백엔드가 카카오 왕복 후 `{redirect_uri}?accessToken=...&refreshToken=...&profileImageUrl=...` 로 리다이렉트.
- **Access Token 24시간 / Refresh Token 14일.** Access는 클라이언트만, Refresh는 클라이언트+DB 저장.
- 콜백 처리: `src/pages/oauth-callback` 이 쿼리에서 토큰을 꺼내 `setSession` → 로그인 상태 전환.
- `profileImageUrl`(카카오 프로필 사진)은 **로그인 직후** 이미 클라이언트에 들어온다. (폼 사진 미리보기에 재사용)

### 3-2. 동적 복귀(redirect_uri)

백엔드 개발 서버는 하나인데 프론트는 로컬(`localhost:5173`)과 배포(`vercel.app`) 두 곳. 로그인 시작 시 현재 origin을 `redirect_uri`로 넘겨 **로그인을 시작한 곳으로 그대로 복귀**시킨다. (배포의 SPA fallback은 `vercel.json`의 rewrite로 해결 — 자세한 내역은 `docs/troubleshooting/kakao-login-redirect.md`)

### 3-3. 상태 관리

- `src/contexts/AuthContext.tsx` 가 토큰 저장소를 구독해 `isLoggedIn` 을 동기화하고, `userApi.me()`로 사용자 정보를 가져온다.
- `loginWithKakao(returnTo?)` 는 로그인 후 돌아올 경로를 `sessionStorage`에 보관 → 콜백에서 복원. (지인이 작성 중이던 폼으로 되돌리는 데 사용)

### 4-1. 홈 (랜딩) — `/`

서비스 가치를 전달하고 핵심 액션으로 유도하는 마케팅 페이지. (`src/pages/home`)

**섹션 구성**

1. **HeroSection** — "좋은 사람은, 가까운 사람이 더 잘 아니까." 메인 카피 + 스택형 카드 미리보기 애니메이션. CTA 2개: `소개 둘러보기`(탐색), `친구 등록하기`.
2. **HowItWorksSection** — 3단계 안내: ① 프로필을 등록한다 ② 카드를 탐색한다 ③ 연결 요청을 보낸다.
3. **CardUXSection** — 카드 UX 소개.
4. **TrustSection** — 신뢰(지인 기반) 메시지.
5. **CTASection** — 하단 행동 유도.

**기능**

- `친구 등록하기` 클릭 시 **로그인 상태면** `/friends`로 이동하며 친구 추가 플로우 자동 트리거(`state.triggerAddFriend`), **비로그인이면** 로그인 모달 노출.
- 헤더는 `variant="landing"` — 스크롤 시 그림자/보더가 생기는 고정 헤더.

### 4-2. 로그인 — `/login`

(`src/pages/login`)

- 카카오 단일 로그인 버튼(`카카오로 시작하기`)만 제공. 클릭 시 `loginWithKakao()`.
- 이미 로그인 상태면 홈으로 `replace` 리다이렉트.

---

### 4-3. OAuth 콜백 — `/oauth2/callback`

(`src/pages/oauth-callback`)

- 백엔드 리다이렉트 URL의 `accessToken`/`refreshToken`/`profileImageUrl`을 읽어 `setSession` 호출.
- `handledRef`로 **중복 실행 방지**(React Strict Mode 대비).
- 로그인 전에 보던 경로(`POST_LOGIN_REDIRECT_KEY`)가 있으면 그곳으로, 없으면 탐색(`/explore`)으로 이동.
- 그 경로가 폼(`/form/:madamId`)이었다면 `FORM_KAKAO_AUTHED_KEY`에 표시 → 폼이 "이 세션에서 본인인증 완료"로 인식.
- 토큰이 없으면 로그인 페이지로.

### 4-4. 탐색 (Explore) — `/explore`

다른 주선자들이 등록한 지인 카드를 스와이프로 둘러보고 연결을 요청하는 **핵심 발견 화면**. (`src/pages/explore`)

**비로그인 상태**

- 카드 대신 "로그인하고 둘러보세요" 안내 + 카카오 로그인 버튼. (카드 상세·요청은 로그인 후)

**로그인 상태 — 주요 기능**

- **성별 필터**: 전체 / 남성 / 여성. URL 쿼리(`?gender=male|female`)와 동기화해 새로고침·공유에도 유지. (`useSwipeCards`가 `profileApi.listPublic({ gender })` 호출)
- **스와이프 카드 스택**(`SwipeCardStack`): 추천 카드를 한 장씩 넘긴다. 넘긴 카드 id는 로컬 상태(`swipedIds`)로 걸러 다시 안 보이게 한다.
- **튜토리얼 카드**(`TutorialCard`): 최초 진입 시 조작법 안내. 쿠키(`madam_tutorial_seen`)로 1회만 노출.
- **로딩/에러/빈 상태**: 스켈레톤(`LoadingCard`), 에러 카드(`ErrorCard`, 네트워크/서버 구분 + 재시도), "카드를 탭하면 자세히 볼 수 있어요" 힌트.
- **상세 모달**(`DetailModal`): 카드 탭 시 사진·소개·취미·MBTI 등 전체 정보. 비로그인 탭 시 로그인 모달.
- **연결 요청 모달**(`ContactRequestModal`): "소개할 내 친구" 드롭다운에서 내 등록 친구를 고르고 한마디 메시지를 적어 연결 요청 전송. 등록된 친구가 없으면 먼저 등록하라고 안내. 전송 성공 시 "상대 주선자가 승인하면 서로의 연락처가 공유돼요" 확인 화면.

> 데이터: `PublicProfile`(연락처 미포함 공개 목록)을 화면용 `Profile`로 변환. 취미 문자열은 콤마/슬래시 기준으로 태그 배열화.

### 4-5. 내 친구 (Friends) — `/friends`

주선자가 자신이 등록·소개하는 지인(=내 데이팅 프로필)을 관리하는 화면. (`src/pages/friends`)

**탭 구조** (모바일 상단 탭 / 데스크탑 좌측 사이드바)

- **등록된 친구**: 승인 완료(`PUBLISHED`)되어 카드로 공개된 친구.
- **승인 대기**: 폼 제출은 됐으나 아직 공개 전인 친구. (배지로 개수 표시)

**친구 추가 / 초대 (숏링크 공유)**

- `친구 추가` → `acquaintanceApi.myForm()`로 **내 폼 숏링크(formUrl)** 를 받아 클립보드 복사.
- **웹**: 토스트로 "복사했어요" 안내만.
- **앱(모바일)**: 바텀시트(`FriendInviteSheet`)로 공유 채널 노출 — 링크 복사 / 문자(SMS) / 시스템 공유 시트(`navigator.share`). 환경별 폴백 처리(데스크탑·미지원 기기는 복사로 대체).

**친구 카드 / 상세** (`FriendCard`, `FriendDetailModal`)

- 카드: 사진(없으면 파스텔 폴백) + 이름·나이 + 소속/직업.
- 상세(모바일은 드래그로 닫는 전체화면 시트): 다음 액션 제공
  - **삭제**(프로필 제거)
  - **비활성화/활성화**(매칭 노출에서 잠시 빼기 ↔ 다시 공개 — `visibility` PRIVATE/PUBLIC 전환)
  - **승인/거절**(폼 제출 친구 등록 확정 — 일부는 백엔드 명세 대기)
  - **폼 재수정 요청**: 새 숏링크를 복사해 친구에게 다시 보내 정보 갱신 유도.

> 데이터: "내 친구" = 내 데이팅 프로필. 목록(`GET /api/profiles`)은 필드가 적어, 각 항목 상세(`GET /api/profiles/{id}`)를 추가로 받아 카드 정보를 채운다. 목록은 60초마다 폴링.

### 4-6. 등록 폼 (Register) — `/register`, `/form/:madamId`

지인의 프로필 정보를 입력받는 **7단계 마법사 폼**. 같은 페이지 컴포넌트를 두 경로가 공유한다. (`src/pages/register`)

**두 가지 진입**

- `/register` — 직접 진입(주로 내부/테스트용).
- `/form/:madamId` — **지인이 주선자의 숏링크로 진입하는 정식 경로.** `madamId`는 제출 시 path variable.

**접근 게이트 (숏링크 진입 시)** — 순서대로 통과해야 작성 가능:

1. **카카오 본인인증 강제**: 이 폼을 이번 세션에 인증하지 않았으면 카카오 로그인으로 자동 이동. (본인 인증 + 주선자 중복가입 방지 목적)
2. **세션 동기화**(스피너)
3. **숏링크 유효성 확인**: `GET /form/{madamId}`. 만료/오류면 "유효하지 않은 링크예요" 안내.

**7단계 입력** (`STEP_LABELS`: 사진 → 기본 정보 → 소속 → MBTI → 소개 → 연락처 → 미리보기)

1. **사진**: 카카오 프로필 사진 사용 여부를 모달로 선택("이 사진으로 할게요" / "직접 올릴게요"). 직접 업로드는 JPG·PNG·WEBP·GIF, 10MB 이하 제약.
2. **기본 정보**: 이름 → 나이(19세 이상) → 성별. 한 스텝 안에서 입력을 하나씩 위로 쌓아 노출(점진적 공개 + revealStage).
3. **소속**: 학생 여부 분기 → 학교·학과 / 직업.
4. **MBTI**: 4축 2지선다(E/I·N/S·T/F·J/P)로 분해 입력.
5. **소개**: 취미 다중 선택(+커스텀 취미 추가) → 한 줄 소개.
6. **연락처**: 전화번호(중복 방지·비공개, 010 형식 검증) + 공개 연락처(인스타/카카오 중 하나 필수).
7. **미리보기**(`CardPreview`): 실제 카드 모양으로 입력값을 확인 후 제출.

**입력 UX 디테일**

- Enter 키로 다음 단계 진행(IME 한글 조합 중 Enter, textarea 줄바꿈은 제외).
- 진행 막대 + `현재/총 단계` 표시.
- 되돌아왔을 때 이미 채운 앞 단계는 펼친 채로 시작.

**제출 흐름** (`폼_인증.pdf`, `form-photo-guide.md`, `kakao-photo-flow.md` 기준)

1. `POST /form/{madamId}` — body는 입력값, `madamId`는 path, 지인의 accessToken은 `Authorization` 헤더로 분리 전송. 응답으로 `uploadToken` 확보.
2. `useKakaoPhoto=true`면 서버가 카카오 프로필 사진를 `displayOrder=0`으로 자동 등록(클라이언트 추가 작업 없음).
3. 직접 올린 사진이 있으면 `POST /form/{uploadToken}/photos`로 별도 업로드.
4. 사진 단계만 재시도할 때 `uploadTokenRef`로 텍스트 중복 제출 방지.
5. 완료 화면: "제출 완료! 주선자의 승인을 기다리고 있어요."

> **카카오 프로필 사진 미리보기의 진실**: 사진 URL은 폼 제출 응답이 아니라 **로그인 직후 redirect 쿼리(`profileImageUrl`)** 로 이미 받아둔 값으로 띄운다. 저장되는 사진은 **폼을 작성하는 지인 본인**의 프로필 사진이다(주선자의 것이 아님).

### 4-7. 요청함 (Requests) — `/requests`

매칭 요청을 관리하는 화면. 가장 정교하게 UX가 설계된 영역. (`src/pages/requests`, 설계 근거: `docs/UX/요청함-매칭-카드-및-상세-UX-설계.md`)

**탭 구조** (탭마다 사용자의 할 일(JTBD)이 다름)

| 탭            | 사용자가 결정/확인할 것                           | 비고                                             |
| ------------- | ------------------------------------------------- | ------------------------------------------------ |
| **받은 요청** | "내 지인에게 이 사람을 붙여줄까?" → 수락/거절     | 대기 건수 배지. 30초 폴링(배지 때문에 항상 조회) |
| **보낸 요청** | "상대 주선자가 응답했나?" → 진행 상태 확인 / 취소 | 해당 탭 활성 시에만 조회                         |
| **성사됨**    | "연락처 확인"                                     | 인라인으로 연락처 노출                           |

**리스트 카드** (`MatchingCard`)

- 두 사람을 겹친 두 아바타로 보여주지 않고 **상대(counterpart) 한 얼굴 + 관계 텍스트 한 줄**(`상대 ♥ 내 지인`)로 단일화. (Hick's Law — 비교 대상 축소)
- 사진 없으면 이름 첫 글자 + 고정 파스텔 폴백(`ProfileAvatar`).
- 상태 배지(모노크롬) + 응답 기한(24시간 미만이면 코랄로 강조).

**상세 모달** (`MatchingDetailModal`) — 점진적 공개 3단 구조

1. 관계 헤더("OO님이 제안했어요" + 상태 배지 + 진행 스테퍼 `제안·수락·성사`)
2. 상대 지인(사진 크게 — 결정의 주인공)
3. 내 지인(접힌 보조 정보, 받음/보냄 관점에 따라 문구 분기)
4. 메시지·기한·연락처(성사 시 인라인으로 바로 노출 — 모달 위 모달 회피)
5. 액션(수락/거절 · 요청 취소 — 긍정 경로를 더 넓고 진하게)

**액션** (`useMatchingActions`)

- 수락(`accept`) / 거절(`reject`) / 취소(`cancel`). 변이 중인 카드만 로딩·비활성화(`busyId`).
- 성공 시 매칭 목록 전체 무효화로 즉시 갱신.

**카피 원칙(토스 톤)**: 내부 용어를 지우고 관계 중심 2인칭으로. 보낸 요청 탭에서 내가 쓴 메시지는 "나님의 한마디" 대신 "내가 남긴 한마디"로 분기(`isMe()`).

> **백엔드 의존 TODO**: 매칭 응답에 `photoUrl`(현재 이니셜 폴백), 뷰어 기준 방향 식별자(현재는 탭으로 추론, 성사됨 탭은 requester를 상대로 가정)가 추가되면 정밀화 예정.

## 5. 공통 시스템

### 5-1. 글로벌 네비게이션

- **AppHeader** (`src/components/AppHeader.tsx`): 로고 + (로그인 시) 탐색/내 친구/요청함 + 알림 종 + 계정 팝오버(로그아웃). 데스크탑 상단 네비, 모바일은 하단 탭으로 대체(`SIDEBAR_ENABLED=false`로 햄버거 드로어는 보존하되 비활성).
- **BottomNav** (모바일 전용): 내 친구 / **탐색(가운데)** / 요청함. 핵심 액션인 탐색을 중앙 배치.

### 5-2. 알림 시스템

- **NotificationBell**: `GET /api/notifications` 30초 폴링 + 읽지 않음 배지. 패널은 뷰포트 우측 고정(헤더 높이 변화·스크롤에 대응).
- **알림 타입별 카피**(`notificationMeta`): 폼 제출 / 매칭 요청·수락·거절·성사 / 당사자 동의 / 본인인증 완료 / 차단 / 기한 만료 등.
- **알림 라우팅**: `form_submitted` 알림은 지인 폼 승인 모달(`AcquaintanceReviewModal`)을 바로 열고, 그 외는 요청함으로 이동. (form_submitted 알림의 `matchingId` 필드엔 실제로 `acquaintanceId`가 담김)
- **폼 승인 모달**(`useAcquaintanceReview`): 주선자가 지인 폼 상세를 보고 승인(`approve`)/거절(`reject`). 성공 시 알림·내 프로필 목록 캐시 무효화.

### 5-3. 기타 공통

- **LoginModal**: 비로그인 사용자가 보호된 액션 시도 시 노출하는 로그인 유도 모달.
- **Loading / 스켈레톤**: 페이지별 로딩 폴백, 카드/친구/매칭 스켈레톤.
- **QueryProvider**: TanStack Query 클라이언트 + Devtools.

## 6. API 레이어 개요

`src/lib/api` — `mydestiny-api-spec` 5장 기준. 공통 응답 래퍼(`{success, message, data}`)를 `unwrap`으로 벗겨 `data`만 반환. (`client.ts`, `endpoints.ts`, `models.ts`, `types.ts`, `tokenStore.ts`, `queryKeys.ts`)

**도메인별 API 묶음**

| 묶음              | 주요 기능                                                       |
| ----------------- | --------------------------------------------------------------- |
| `authApi`         | 토큰 갱신(refresh), 로그아웃                                    |
| `userApi`         | 내 정보(me), 닉네임 변경                                        |
| `registrantApi`   | 등록자(주선자) 목록·상세·팔로잉/팔로워·bio                      |
| `followApi`       | 팔로우/언팔로우/상태                                            |
| `formApi`         | 폼 유효성(validate)·제출(submit)·사진 목록/업로드/교체          |
| `acquaintanceApi` | 초대 링크(invite)·내 폼(myForm)·상세·승인/거절                  |
| `profileApi`      | 프로필 생성·내 목록·공개 목록·상세·수정·삭제·사진·공개범위·신고 |
| `invitationApi`   | 당사자 승인 흐름(info·profile·consent·approve·reject)           |
| `cardApi`         | 카드 목록/상세                                                  |
| `matchingApi`     | 매칭 생성·보낸/받은/성사 목록·수락/거절/취소·연락처             |
| `consentApi`      | 당사자 동의 대기·승인/거절                                      |
| `phoneApi`        | 휴대폰 인증 발송/검증                                           |
| `notificationApi` | 알림 목록·읽음 처리                                             |
| `blockApi`        | 차단/해제                                                       |
| `adminApi`        | 관리자(프로필 심사·신고·매칭 관리)                              |

> 개발 환경에서는 `.env.local`로 dev 토큰을 주입할 수 있다. (BASE_URL 기본값: `https://fixlog.art/destiny`)

## 7. 현재 구현 상태 & 백엔드 갭

문서로 명시된, 기획 플로우와 현행 API 명세의 차이:

- **숏링크 생성 → 폼 게이팅**: 폼 진입 카카오 인증, "주선자은 등록할 수 없습니다" 중복가입 차단, 제출 시 인가코드 동봉 등은 **백엔드 API 대기**. (현재 프론트는 옵션 B — 세션 카카오 인증 강제 + `GET /form/{madamId}` 유효성 확인으로 임시 동작) — 상세: `docs/apis/FORM_FLOW.md`
- **요청함 매칭 응답**: `MatchingProfileRef.photoUrl`, 뷰어 방향 식별자 추가 요청됨 — 도착 전까지 이니셜 폴백/탭 추론.
- **친구(프로필) 승인/거절 엔드포인트**: 일부 명세 미확정으로 `useFriends`에서 TODO 처리.
- **로그인 동적 복귀**: 프론트는 `redirect_uri` 전달 완료, 백엔드 화이트리스트 검증·CORS 추가는 요청서(`docs/apis/backend-login-redirect-request.md`)로 전달.
