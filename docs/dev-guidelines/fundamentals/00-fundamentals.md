# 프론트엔드 코드 운영 규칙

아래 8가지를 팀 기본 합의로 둔다.  
각 규칙은 가독성, 예측 가능성, 응집도, 결합도라는 4가지 축에 대응한다.

---

**page는 조립만 한다.**  
인증, 권한, API 세부사항, 정규화 로직을 `index.tsx`에 길게 두지 않는다.  
로직은 `pages/<name>/hooks/`로 내리고, 페이지는 구조를 읽히게 한다.

---

**페이지 전용 코드는 페이지 폴더 안에 둔다.**  
`components/`, `hooks/`, `utils/`는 여러 페이지에서 실제로 쓰일 때만 올린다.  
한 페이지에서만 바뀌는 코드는 `pages/<name>/` 안에 colocate한다.

---

**services는 서버 통신만 담당한다.**  
API 함수에서는 토스트, 상태 변경, 페이지 이동을 하지 않는다.  
그런 흐름은 페이지 hook에서 조합한다.

---

**공통 폴더는 정말 공용일 때만 올린다.**  
"2번 썼다"가 기준이 아니라 "앞으로도 함께 바뀔 가능성이 높다"여야 한다.

---

**서버 조회 Hook 반환 규칙을 통일한다.**  
TanStack Query 기반 조회 Hook은 query 객체 반환으로 통일한다.  
`data`, `isLoading`, `error`, `refetch`를 일관되게 제공한다.

---

**숫자와 문자열 정책은 이름을 준다.**  
debounce 시간, polling interval, 최대 파일 크기, page size는  
상수나 맵으로 관리한다. 매직 넘버를 코드에 직접 두지 않는다.

---

**중간 전달 prop이 늘어나면 local context를 검토한다.**  
2단계 이상 연속 전달되고 중간 컴포넌트가 prop을 사용하지 않으면  
Context 또는 상태 끌어올리기를 우선 검토한다.

---

**공통화보다 변경 방향을 먼저 본다.**  
요구사항이 조금씩 달라질 모달/리스트/툴바는 처음부터 억지로 합치지 않는다.  
비슷해 보여도 함께 바뀌지 않을 것 같다면 각자 두는 것이 낫다.
