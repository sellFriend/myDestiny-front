# Git 브랜치 규칙

## 1. 목적

- 안정적으로 개발·테스트·배포를 진행하기 위해 브랜치 역할을 명확히 구분한다.
- **배포 기준(main)** 과 **개발 통합 기준(dev)**, **작업 단위 브랜치**, **개인 작업 브랜치** 를 분리한다.

---

## 2. 브랜치 종류 및 역할

### 2.1 `main`

- **배포 기준 브랜치**
- 검증 완료된 변경사항만 머지한다.
- 이 브랜치를 기준으로 배포한다.

**규칙**

- 릴리즈 시점에만 머지를 원칙으로 한다.
- 긴급 수정(Hotfix)이 필요한 경우에는 최소한의 변경만 반영한다.

---

### 2.2 `dev`

- **개발 통합(Integration) 브랜치**
- 개발 및 테스트가 완료된 작업을 순차적으로 머지한다.
- 팀의 "현재 개발 최신 상태"를 대표한다.

**규칙**

- 작업 브랜치에서 기능이 완료되고 테스트를 통과하면 `dev`로 머지한다.
- `dev`는 항상 빌드/기본 테스트가 가능한 상태를 지향한다.

---

### 2.3 `<type>/<short-description>` (작업 브랜치)

- **작업 단위 브랜치**
- 하나의 기능 또는 버그 수정 단위에 대응하는 브랜치.
- 개인 작업 브랜치에서 올라온 작업들을 이 브랜치로 머지한 뒤, 통합 테스트/검증을 수행한다.

**규칙**

- 작업 완료 시점에 `dev`로 머지한다.
- 브랜치는 "작업 범위 내 변경만" 포함한다(스코프 관리).

**명명 예시**

- `feat/login-oauth`
- `fix/duplicate-payment`
- `refactor/document-editor`
- `chore/update-dependencies`

---

### 2.4 `<이니셜>/<브랜치명>` (개인 작업 브랜치)

- **개인 작업용 브랜치**
- 개인이 자유롭게 작업하고 수시로 푸시할 수 있는 브랜치.
- 검증이 끝난 작업은 작업 브랜치로 올려 통합한다.

**규칙**

- 개인 브랜치는 작업 편의용이므로, 직접 `dev`/`main`으로 머지하지 않는다(원칙).
- 브랜치명은 팀에서 식별 가능하도록 간단하고 명확하게 작성한다.

**명명 예시**

- `hs/login-oauth`
- `hs/fix-payment`
- `mh/document-editor`

---

## 3. 권장 작업 흐름(Flow)

1. **작업 범위 확정**
   - 작업 브랜치 생성: `<type>/<short-description>`
2. **개인 작업 브랜치에서 개발**
   - 개인 브랜치 생성: `<이니셜>/<작업명>`
   - 개발/커밋/푸시 반복
3. **개인 브랜치 → 작업 브랜치로 통합**
   - PR/머지 후 기능 단위 테스트/검증 수행
4. **작업 브랜치 → `dev` 머지**
   - 개발 및 테스트 완료 후 `dev`에 통합
5. **릴리즈 시 `dev` → `main` 머지**
   - 릴리즈 범위 확정 및 회귀 테스트 후 `main`에 반영
   - `main` 기준 배포
   - 버전 태그 생성: `v<major>.<minor>.<patch>`

---

## 4. 브랜치 보호/머지 규칙(권장)

> 아래 항목은 프로젝트 성격에 따라 선택 적용한다.

### 4.1 머지 방식

- 기본 원칙: **리베이스 후 머지(Rebase 후 Merge)** 를 사용해 **Git 히스토리를 선형으로 유지**한다.
  - 작업 브랜치의 커밋을 `dev` 최신 기준으로 **rebase**한 뒤 머지한다.
  - 불필요한 merge commit을 줄이고 변경 이력을 선형(linear)으로 관리한다.

**권장 운영 방식**

- PR 머지 시: **Rebase and merge** 옵션 사용
- 로컬에서 rebase 수행 시: 충돌 해결 후 테스트까지 완료한 다음 머지

> 주의: 리베이스는 커밋 해시를 변경하므로, 여러 사람이 동시에 공유하는 브랜치에서는 팀 합의 하에 진행한다.

### 4.2 보호 규칙

- `main`, `dev`는 **직접 push 금지**, PR(리뷰) 기반 머지 권장
- `main` 머지는 릴리즈 담당자가 수행

---

## 5. 요약

| 브랜치 | 역할 |
|--------|------|
| `main` | 배포 기준, 릴리즈 후만 머지 |
| `dev` | 개발 통합, 완료 작업을 순차 머지 |
| `<type>/<description>` | 작업 단위 개발/테스트 브랜치 |
| `<이니셜>/<브랜치명>` | 개인 작업 브랜치 |
