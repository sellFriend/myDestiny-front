# 요청함(매칭) 카드 · 상세 UX 설계

> 대상 화면: `요청함` 페이지의 **받은 요청 / 보낸 요청 / 성사됨** 탭과 그 카드·상세.
> 관련 코드: `src/pages/requests/components/MatchingCard.tsx`, `MatchingDetailModal.tsx`, `ProfileAvatar.tsx`, `utils.ts`

이 문서는 요청함 카드/상세를 재설계하며 내린 결정과 그 **UX적 근거**를 기록한다. "왜 이렇게 했는가"를 남겨, 이후 수정 시 같은 함정을 반복하지 않기 위함이다.

---

## 0. 이 화면의 본질 — 탭마다 '할 일(JTBD)'이 다르다

매칭 1건은 항상 **두 사람(지인 A·B)** 과 **두 주선자(제안한 쪽·받은 쪽)** 으로 이뤄진다. 그러나 내가 봐야 하는 핵심은 탭마다 다르다.

| 탭 | 내가 결정/확인할 것 | 핵심 인물 |
| --- | --- | --- |
| 받은 요청 | "내 지인에게 **이 사람**을 붙여줄까?" → 수락/거절 | 상대 지인(새로 평가할 사람) |
| 보낸 요청 | "상대 주선자가 응답했나?" | 진행 상태 |
| 성사됨 | "연락처 확인" | 두 사람 모두 |

> **설계 출발점:** "받은 요청의 상대 지인 한 명"이 정보 설계의 주인공이다. 여기서부터 리스트·상세가 자연스럽게 풀린다.

---

## 1. 리스트 카드 — '두 얼굴'이 아니라 '한 얼굴 + 관계 텍스트'

### 문제
초기 디자인은 두 사람을 **겹친 두 원형 아바타**로 보여줬다. 이는 단순히 색이 충돌하는 문제가 아니라,
리스트에서 두 얼굴이 서로 경쟁해 "누가 누구인지" 인지 비용이 드는 문제였다.

### 결정
- 리스트의 시각 초점은 **상대(counterpart) 한 명의 사진**으로 단일화한다.
- 관계(누구 ↔ 누구)는 사진이 아니라 **텍스트 한 줄**(`상대 ♥ 내 지인`)이 책임진다.
- 사진이 없으면 이름 첫 글자 + 고정 파스텔로 폴백한다. (`ProfileAvatar`)

### 근거 (UX 법칙)
- **Hick's Law** — 동시에 비교할 대상을 줄여 결정 부담을 낮춘다. 얼굴 하나 = 즉시 인지.
- **Aesthetic-Usability Effect** — 얼굴 하나가 깔끔하고 신뢰감 있게 읽힌다.
- **Jakob's Law** — 사람 + 이름 + 관계 라인은 사용자가 이미 익숙한 리스트 패턴.

> 결과적으로 처음의 "겹침/색 충돌" 고민은 단일 초점으로 바꾸자 **문제 자체가 사라졌다.**

## 2. 상세보기 — 점진적 공개(Progressive Disclosure) 3단 구조

두 개의 풀 프로필을 동등하게 나란히/캐러셀로 보여주면 **둘 다 100%가 되어 결국 둘 다 0%** 가 된다.
대신 결정에 필요한 정보를 위계에 따라 단계적으로 드러낸다.

```
① 관계 헤더  : "소개팅요정님이 제안했어요" + 상태 배지 + 진행 스테퍼
② 상대 지인  : 사진 크게 + 이름/성별 (결정의 주인공)
③ 내 지인    : 접힌 보조 정보 (받음/보냄 관점에 따라 문구가 다름 / 펼치기)
④ 메시지·기한·연락처(성사 시 인라인으로 바로 노출)
⑤ 액션       : 수락/거절 · 요청 취소
```

### 받은 요청 vs 보낸 요청 — 같은 골격, 다른 관점

상세 골격은 같지만 **내 지인이 "받는 쪽"인지 "내보낸 쪽"인지**가 달라 문구를 분기한다.

| 영역 | 받은 요청 | 보낸 요청 |
| --- | --- | --- |
| 헤더 | `OO님이 제안했어요` | `OO님에게 제안했어요` |
| 상대 섹션 라벨 | `이런 분을 소개받았어요` | `이 분에게 소개했어요` |
| 내 지인 보조 | `내 지인 OO님에게 들어온 제안이에요` | `내 지인 OO님을 소개한 제안이에요` |

### 성사됨 — 연락처는 버튼이 아니라 상세 안에서 바로

별도 '연락처 보기' 모달(모달 위 모달)을 띄우지 않고, 상세 본문 하단에 연락처를 **인라인으로 펼쳐** 보여준다.
탭 수를 줄이고(Doherty), 맥락(누구의 연락처인지)을 유지한다. 성사된 인연은 두 사람이 동등하므로 내 지인도 기본 펼침.

### 각 영역의 근거
- **① 헤더가 질문 3개를 한 번에 답한다** — "어떤 주선자가 제안했나 / 내가 받았나 보냈나 / 지금 어느 단계인가".
  - *Miller's Law(청킹)* · *Law of Proximity·Common Region* — 관계 정보를 한 덩어리로 묶는다.
  - *Goal-gradient* — 3단계 스테퍼(제안 → 수락 → 성사)로 진행률을 보여 막판 행동을 유도한다.
- **② 상대 지인을 크게** — 이미 아는 내 지인보다, 새로 평가할 상대가 먼저다.
  - *Serial Position Effect* — 중요한 것을 앞에.
- **③ 내 지인은 접어둠** — 이미 아는 사람이므로 기본은 감추고, 필요할 때만 펼친다.
  - *Progressive Disclosure* · *Hick's Law* — 한 번에 보이는 정보를 줄인다.
- **⑤ 액션** — 긍정 경로(수락)를 더 넓고 진하게.
  - *Von Restorff Effect* · *Fitts's Law* — 핵심 액션을 도드라지게, 탭 영역을 크게.

## 3. 기존 '카드 디자인' 재사용 / '넘기기(스와이프)' 여부

### 결정
- **카드의 비주얼 언어는 재사용**한다 — 사진영역 + 정보영역, 파스텔 정체성, `rounded-block`.
  - *Jakob's Law* — 둘러보기(Explore) 카드에서 학습한 패턴을 그대로 활용, 프로젝트 일관성 유지.
- **숨은 스와이프 캐러셀은 쓰지 않는다.**
  - 두 번째 사람을 숨기면 "더 있나?"라는 불확실성이 생기고(발견성↓), 두 사람이 양자택일처럼 보인다.
  - 매칭은 양자택일이 아니라 **둘의 관계**다.
- 둘을 전환해야 한다면 숨은 제스처가 아니라 **라벨된 컨트롤**(접힘/펼침 등 명시적 UI)로 한다.
  - 토스 원칙: "영리한 숨은 제스처보다 명확하게 보이는 컨트롤."

> 현재 구현은 ③ 접힘 구조로 충분해 별도 토글조차 두지 않았다 — 더 단순한 쪽을 택했다.

## 4. UX 라이팅 (토스 톤)

내부 용어(requester/target/주선자)를 화면에서 지우고, **관계 중심 · 2인칭 · 안심**으로 쓴다.

| 위치 | 카피 |
| --- | --- |
| 헤더(받음) | `소개팅요정님이 제안했어요` |
| 헤더(보냄) | `인연메이커님에게 제안했어요` |
| 방향 보조(받음) | `내 지인 이준호님에게 들어온 제안이에요` |
| 방향 보조(보냄) | `내 지인 박지민님을 소개한 제안이에요` |
| 상대 소개 | `이런 분을 소개받았어요` / `이 분에게 소개했어요` |
| 메시지 작성자 | `소개팅요정님의 한마디` — 내가 쓴 경우(보낸 요청)엔 `내가 남긴 한마디`로 (`나님` 금지) |
| 스테퍼 | `제안 · 수락 · 성사` (전문용어 배제) |
| 액션 | `수락하기` / `요청 취소` (성사 연락처는 인라인) |
| 기한 | `2일 안에 응답해 주세요` (24시간 미만이면 코랄로 강조) |

> **주의:** 매칭 메시지 작성자는 항상 제안한 주선자(requester)이다. 보낸 요청 탭에선 그게 '나'이므로
> `나님의 한마디` 같은 어색한 표기를 피하려 `isMe()`로 판별해 `내가 남긴 한마디`로 바꾼다. 카드·상세 공통.

### 라이팅 원칙
- 명령조(`보내주세요`)보다 권유·사실 전달.
- 한 카드/화면에 하나의 핵심 메시지.
- 민감 정보(얼굴·연락처)에는 안심 문구를 곁들여 신뢰를 쌓는다.

## 5. 상태·기한 표현

- **상태 배지**: 모노크롬 유지(`statusMeta`). 진행=연한 회색, 완료/성사=검정, 종료=옅은 회색.
- **하트 채움으로 성사 표현**: 두 이름 사이 하트가 평소엔 빈 회색, 성사 시 코랄 채움. (Von Restorff · Peak-End)
- **기한 강조(goal-gradient)**: `remainingMeta`가 24시간 미만이면 `urgent`로 코랄 강조해 막판 응답을 유도.
- **진행 스테퍼(goal-gradient)**: `matchingSteps`가 상태를 사용자 관점 3단계(제안·수락·성사)로 환산.
  '당사자 동의'(`CONSENT_*`)는 **백엔드 내부 단계**라 별도 노출하지 않고 수락~성사 사이로 합친다.
  (동의 액션 화면 자체가 미구현이라, 보여줘도 사용자가 갈 곳이 없다.) 거절/취소/만료는 terminal 메시지로 대체.

## 부록 — 적용한 UX 10대 법칙 요약

| 법칙 | 적용 지점 |
| --- | --- |
| Jakob's Law | 카드 비주얼 언어 재사용, 익숙한 리스트 패턴 |
| Hick's Law | 리스트 단일 초점, 내 지인 접힘 |
| Miller's Law | 상세 헤더의 정보 청킹 |
| Law of Proximity / Common Region | 관계 정보를 헤더 한 덩어리로 |
| Serial Position | 상대 지인을 먼저, 액션을 마지막에 |
| Von Restorff | 수락 버튼·성사 하트 강조 |
| Fitts's Law | 큰 탭 영역의 액션 버튼 |
| Aesthetic-Usability | 단일 얼굴·여백·일관 톤 |
| Goal-gradient | 기한 강조 + 진행 스테퍼 |
| Peak-End | 성사 순간의 코랄 하트·축하 톤 |
