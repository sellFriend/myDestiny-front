# 토스 UX 라이팅 원칙과 우리 카피 가이드

> 목적: My Destiny가 "토스 톤"이라고 부르는 글쓰기 기준을 명확히 한다.
> 토스의 철학·보이스앤톤·8가지 라이팅 원칙을 정리하고, **각 원칙을 우리 화면 카피에 어떻게 적용했는지** 실제 문구로 보여준다.

## 0. 왜 토스 톤인가

My Destiny는 **연애·소개**라는 민감하고 사적인 맥락을 다룬다. 사용자는 자기 친구를 카드로 만들고, 모르는 사람에게 연결을 제안한다. 이때 말투가 딱딱하거나(행정 공문 톤) 부담을 주면(강요·과장) 쉽게 위축된다.

그래서 우리는 토스의 라이팅 철학을 차용한다:

> **"인간화된 말하기"** — 정보를 정확히 전달하되, 사람이 사람에게 말하듯 친근하고 안심되게.

토스의 코어밸류 다섯 가지 — **명확한 · 간결한 · 친근한 · 존중하는 · 공감하는** — 를 우리 카피의 합격 기준으로 삼는다.

## 1. 보이스 & 톤 기본 규칙

| 규칙 | 내용 | 우리 적용 |
| --- | --- | --- |
| **해요체 통일** | 상황·맥락 불문 모든 문구를 '해요체'로. | `"연락 요청을 보내려면 로그인이 필요해요."` (전 화면 일관) |
| **능동형 우선** | 수동형은 꼭 필요할 때만. | `"대표 사진을 올려주세요"` (○) / "사진이 업로드되어야 합니다" (×) |
| **긍정형 우선** | 부정·금지보다 가능·안내로. | `"둘 중 하나만 입력해도 충분해요."` |
| **2인칭 관계 중심** | 내부 용어 대신 "OO님" 관계 문장. | `"소개팅요정님이 제안했어요"` (requester/target 같은 용어 금지) |

## 2. 토스의 8가지 라이팅 원칙 + 우리 적용

토스는 A/B 테스트로 쌓은 위닝 전략을 8개 원칙으로 정리했다. 각 원칙을 체크리스트처럼 쓴다.

### 1) Predictable hint — 예측 가능한 힌트
**물음:** "다음 화면에서 무슨 일이 일어날지 예상할 수 있는가?"
다음 단계를 미리 알려 불안을 줄인다.

- 우리 적용: 연결 요청 성공 화면에서 `"상대 주선자가 승인하면 서로의 연락처가 공유돼요"` — 지금 보낸 행동의 **다음 결과**를 미리 알려준다.
- `"매칭이 성사되면 상대에게 공개돼요."` — 입력한 정보가 언제 누구에게 보이는지 예고.

### 2) Weed cutting — 잡초 제거
**물음:** "의미 없는 단어를 다 제거했는가?"
모바일에서 빠르게 스캔되도록 기능 없는 단어를 지운다.

- 우리 적용: `"기본 정보를 알려주세요"` — "고객님의 기본적인 정보를 입력해 주시기 바랍니다" 같은 군더더기를 쳐낸 형태.
- 버튼은 한 동작 = 짧은 동사구: `"이 사진으로 할게요"`.

### 3) Remove empty sentences — 빈 문장 제거
**물음:** "한 화면에서 반복·불필요한 설명을 지웠는가?"
제목에서 이미 말한 걸 본문에서 다시 설명하지 않는다.

- 우리 적용: 제목 `"나를 소개해주세요"` 아래 본문은 같은 말을 반복하지 않고 **방법**만 덧붙인다: `"어떤 사람인지 한두 문장으로 편하게 적어주세요."`

### 4) Focus on key message — 핵심 메시지 집중
**물음:** "지금 정말 필요한 정보만 말하고 있는가?"
기술적 맥락보다 사용자가 당장 결정할 것만.

- 우리 적용: 사진 제약 안내는 형식·용량 **두 가지만** 명확히. `"JPG·PNG·WEBP·GIF 이미지만 올릴 수 있어요."`, `"사진 용량은 10MB 이하만 가능해요."` (인코딩·해상도 같은 부가 정보는 생략)

### 5) Easy to speak — 말하기 쉬운 표현
**물음:** "이해하기 어려운 용어·문어체를 안 썼는가?"
일상 구어체로, 전문 용어를 줄인다.

- 우리 적용: MBTI 축조차 코드(E/I…)가 아니라 일상어로 표현. `"함께 있을 때 충전돼요"` / `"혼자일 때 충전돼요"`, `"논리로 판단해요"` / `"마음으로 공감해요"`, `"현실을 중시해요"` / `"가능성을 상상해요"`.
- "마담/주선자" 같은 내부 호칭도 화면에서는 친근한 닉네임(`OO님`)으로 노출.

### 6) Suggest over force — 강요가 아닌 제안
**물음:** "특정 행동을 강요하거나 공포감을 주지 않는가?"
선택권을 존중하는 중립적 톤.

- 우리 적용: `"간단한 소개 메시지를 남겨보세요"` — '~하세요'(명령)보다 '~보세요'(권유). "안 쓰면 매칭 안 됨" 같은 압박 없음.
- `"둘 중 하나만 입력해도 충분해요."` — 부담을 덜어주는 허용형 문장.

### 7) Universal words — 보편적 언어
**물음:** "모두가 이해하고, 누구에게도 무해한가?"
특정 집단만 아는 유행어·은어·밈을 피한다.

- 우리 적용: 카피에 밈·유행어를 쓰지 않는다. 나이·성별·소속처럼 누구에게나 같은 의미인 단어로 안내. `"소속을 알려주세요"`.

### 8) Find hidden emotion — 숨은 감정 찾기
**물음:** "사용자의 감정에 공감했는가?"
정보 전달을 넘어 감정에 닿는다.

- 우리 적용: 자기소개 입력에서 `"어떤 사람인지 한두 문장으로 편하게 적어주세요."` 의 **"편하게"** — 잘 써야 한다는 부담을 알아채고 덜어준다.
- 성사·축하 같은 절정 순간에는 정보 공지를 넘어 기쁨을 함께 표현한다. ([Peak-End Rule](./UX-법칙-적용-가이드.md#8-peak-end-rule--마지막-순간을-설계한다)과 연결)

## 3. 우리만의 추가 규칙

토스 원칙 위에, My Destiny의 맥락(소개·관계)에서 더한 규칙:

- **내부 도메인 용어 금지어 목록**: `requester`, `target`, `madam`, `acquaintance`, `consent` 등은 화면 카피에 **절대 노출하지 않는다.** 모두 "OO님 / 친구 / 상대" 같은 관계어로 치환.
- **화자 분기**: 같은 메시지라도 보는 사람 기준으로 주어를 바꾼다. 내가 쓴 글은 `"내가 남긴 한마디"`, 상대가 쓴 글은 `"OO님의 한마디"`(`isMe()`).
- **에러는 "원인 + 다음 행동"**: 무엇이 잘못됐는지만 말하지 않고 회복 경로를 같이 준다.
  - `"유효하지 않은 폼 링크예요. 마담에게 링크를 다시 받아 주세요."`
  - `"제출에 실패했어요. 잠시 후 다시 시도해 주세요."`
- **로그인 게이트 문구는 "이유 + 행동"**: `"카드 상세 보기와 연락 요청은 로그인 후 이용할 수 있어요."`, `"친구를 등록하려면 로그인이 필요해요."`

## 4. 카피 작성 체크리스트

새 문구를 쓰거나 고칠 때 아래를 순서대로 통과시킨다.

- [ ] 해요체인가? 능동·긍정형인가?
- [ ] 내부 용어(requester/target/madam…)가 화면에 새지 않았는가?
- [ ] 잡초(기능 없는 단어)와 빈 문장을 제거했는가?
- [ ] 한 화면에서 한 가지 핵심만 말하는가?
- [ ] 강요·공포·과장 없이 제안하고 있는가?
- [ ] 유행어·은어 없이 누구나 이해하는 보편어인가?
- [ ] 다음 단계를 예측할 힌트가 있는가?
- [ ] (해당 시) 사용자의 부담·불안 같은 숨은 감정에 공감했는가?
- [ ] 에러라면 "원인 + 다음 행동"을 함께 줬는가?
