# My Destiny Dev Guidelines

## 목적

My Destiny의 프론트엔드 코드는 “예쁘게 보이는 코드”보다 “변경하기 쉬운 코드”를 목표로 한다.

우리는 아래 4가지 기준으로 코드를 판단한다.

1. 가독성: 코드를 처음 읽는 사람이 빠르게 이해할 수 있는가
2. 예측 가능성: 이름, 타입, 반환값, 동작이 예상대로 움직이는가
3. 응집도: 함께 바뀌는 코드가 함께 모여 있는가
4. 결합도: 불필요하게 넓은 수정 범위를 만들고 있지 않은가

## My Destiny에서 특히 중요한 맥락

- 탐색(explore), 친구(friends), 요청(requests), 인증(auth)은 서로 다른 도메인이다.
- `src/components`는 전역 재사용 컴포넌트만 둔다.
- `src/contexts`는 전역 상태(인증 등)만 둔다.
- `src/pages/<domain>/components`는 해당 도메인 전용 UI를 둔다.
- `src/pages/<domain>/hooks`는 해당 도메인의 상태·비즈니스 로직을 둔다.
- `src/pages/<domain>/index.tsx`는 화면 조립만 담당한다.

## 기본 원칙

- 분기보다 역할 분리를 우선한다.
- 숨은 부작용을 만들지 않는다.
- 공통화는 “지금 비슷하다”가 아니라 “앞으로도 같이 바뀐다”일 때만 한다.
- Hook/함수/컴포넌트는 이름만 보고도 대략 무슨 일을 하는지 예측 가능해야 한다.
- 함께 수정되는 파일은 같은 디렉토리에 둔다.
